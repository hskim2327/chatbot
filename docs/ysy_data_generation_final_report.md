# RFP 데이터 생성 및 Generation 실험 최종 보고서

작성일: 2026-06-02  
작성 범위: RFP corpus 생성, 데이터 보정, 검색/답변 생성 실험, 모델 비교

## 1. 작업 목표

이번 작업의 목표는 RFP 원문을 RAG 실험에 바로 사용할 수 있는 구조화 corpus로 만드는 것이었다.

처음에는 문서를 청크로 나누는 작업처럼 보였지만, 실제 문제는 더 복잡했다. RFP 문서에는 예산, 기초금액, 평가 기준, 지급 조건, 입찰 자격, 사업 기간, 제출 마감일처럼 서로 다른 역할을 가진 정보가 같은 표나 문단 안에 섞여 있다. 이 역할을 구분하지 못하면 검색이 정답 문서를 가져와도 답변 생성 단계에서 엉뚱한 금액이나 조건을 답하게 된다.

그래서 최종 데이터 설계는 세 가지를 분리하는 방향으로 잡았다.

```text
검색 가능한 본문
판단 가능한 metadata
추적 가능한 원문 근거
```

즉, 숫자만 잘 뽑는 것이 아니라 RFP 문서를 검색 가능한 텍스트, 답변 판단에 필요한 정보, 원문 확인용 근거로 나누는 것이 핵심이었다.

## 2. 전체 작업 흐름

```text
HWP/PDF 원문
→ HWPX/XML 변환
→ 표 row/column 추출과 본문 청킹
→ RFP 도메인 기반 JSONL 구조화
→ Retrieval / Generation 실험
→ 실패 사례 기반 데이터 보정
→ slim corpus 공유
```

이 과정에서 원문에 사업예산, 입찰마감일, 공고번호가 없는 경우에는 나라장터 정보를 별도로 확인해 보강했다. 특히 원문에 없는 예산금액은 33건 보강했다. 이 값은 원문 추출값처럼 처리하지 않고, 외부 확인값이라는 흔적을 남겼다.

## 3. HWPX 변환과 표 추출

원본에는 HWP와 PDF가 섞여 있었다. 일반 텍스트 추출만 사용하면 표의 행/열 구조가 무너지고, `사업예산`, `사업기간`, `입찰참가자격`, `평가기준` 같은 정보가 주변 숫자와 섞이는 문제가 있었다.

HWP를 HWPX로 변환한 뒤 XML 구조를 사용하면 다음 작업이 안정적이었다.

- 표 row/column 단위 추출
- 문단 순서 보존
- 섹션 경로 유지
- 본문과 표의 chunk type 분리
- 검색된 청크에서 원문 저장소로 다시 연결

최종 청크는 아래처럼 검색용 정보와 원문 참조 정보를 함께 가진다.

```python
chunk = {
    "chunk_id": "doc_xxx_table_0021_part_001",
    "chunk_type": "table",
    "content": "[문서: ... | 섹션: ... | 유형: table] ...",
    "source_file": "기관명_시스템 구축사업.hwp",
    "metadata": {...},
    "source_ref": {
        "source_store_id": "src_doc_xxx_table_0021",
        "block_id": "Dxxx_v2_table_0021",
        "part_index": 1
    }
}
```

## 4. JSONL 구조 설계

### 4.1 검수용 corpus와 실행용 corpus 구분

최종 데이터는 두 단계로 나누어 관리했다.

| 구분 | 용도 |
|---|---|
| `chunks_v2_690_full.jsonl` | 검수용. 보정 흔적, 전체 key, 자세한 출처 확인용 |
| `chunks_v2_690.jsonl` | 기본 실행용. Chroma 적재와 검색/답변 생성 실험용 |
| `source_store_v2_690.jsonl` | 검색된 청크의 긴 원문과 표 근거를 다시 확인하는 원문 저장소 |

검수용 corpus는 상위 key 74개, metadata key 91개를 유지했다. 이후 실험용 corpus에서는 검색과 답변 판단에 필요한 핵심 정보만 남겨 상위 key 10개, metadata key 57개로 줄였다. row 수는 106,777개로 유지하면서 파일 크기는 500.05 MB에서 384.42 MB로 줄였다.

### 4.2 최종 slim JSONL top-level key

최종 검색용 `chunks_v2_690.jsonl`의 top-level key는 아래 10개다.

```text
chunk_id
chunk_type
content
doc_id
doc_key
embed_enabled
fact_type
metadata
source_file
source_ref
```

Chroma 적재 시에는 다음처럼 대응된다.

```text
ids       = chunk_id
documents = content
metadatas = metadata
```

### 4.3 metadata 역할

metadata는 검색 결과를 최종 답변에 써도 되는지 판단하기 위한 정보다. 예를 들어 같은 금액이라도 사업예산인지, 평가 기준인지, 입찰 기준 금액인지 구분해야 한다.

대표적인 metadata 역할은 다음과 같다.

| 역할 | 예시 |
|---|---|
| 문서 식별 | 발주기관, 사업명, 원본 파일명, 공고번호 |
| 위치 정보 | 섹션, 블록, 표/본문 구분 |
| 답변 정책 | 최종 답변에 사용할 수 있는 fact인지 여부 |
| 금액 판단 | 사업예산, 후보 금액, 기준 금액, 지급 조건 구분 |
| 일정 판단 | 사업기간, 입찰마감일, 유지보수 기간 |
| 원문 연결 | source store와 연결되는 참조값 |

예산 질문에서는 단순히 숫자가 있는 청크를 쓰지 않고, `fact_type`, `budget_answer_enabled`, `budget_value_role`, `answer_policy`를 함께 확인했다.

```text
project_budget + budget_answer_enabled=True
→ 사업예산 답변에 사용할 수 있음

threshold_budget / payment_terms / notice_base_amount
→ 사업예산 답변 근거로 바로 사용하지 않음
```

## 5. 원문 우선 원칙과 외부 보강

데이터 보정은 아래 순서로 진행했다.

1. 원본 RFP 문서 전체 대조
2. 원문에서 확인되지 않는 경우에만 나라장터 상세 정보 확인
3. 보강값에는 원문 추출값과 다른 표시를 남김

보강 대상은 사업예산금액, 입찰마감일, 공고번호였다. 특히 원문에 사업예산이 없어서 나라장터에서 확인한 예산금액은 33건이다.

이 처리가 중요한 이유는 외부 보강값을 원문에 있던 값처럼 섞으면 근거 충실도 평가에서 문제가 생길 수 있기 때문이다. 따라서 보강값은 답변 생성에는 활용하되, 출처 구분은 남기는 방식으로 처리했다.

## 6. v1 → v2 구조화가 검색에 준 영향

초기 v1은 정제 텍스트를 일정 길이로 나눈 기준선에 가까웠다. 반면 v2는 표 구조, 문서 요약, 예산/기간/제출서류 같은 핵심 후보, 원문 참조를 분리한 구조화 corpus다.

비교 결과는 구조화가 실제 검색 품질에 긍정적으로 작용했다는 것을 보여준다.

| 기준 | 설명 | Hit@5 | MRR@5 | nDCG@5 | Doc Recall@5 | Multi-doc Recall@5 | Latency |
|---|---|---:|---:|---:|---:|---:|---:|
| v1 초기 | KoE5 dense baseline | 0.942 | 0.867 | 0.790 | 0.806 | 0.635 | avg 28.69ms |
| v2 초기 | dense + BM25 RRF + reranker | 0.980 | 0.958 | 0.886 | 0.878 | 0.750 | avg 3263.32ms |
| v2 최고 | dense + BM25 RRF + reranker | 0.990 | 0.959 | 0.9578 | 0.9825 | 0.9792 | P95 3602.69ms |

특히 여러 문서를 함께 찾아야 하는 질문에서 개선 폭이 컸다. 이는 표 추출과 JSON key 구조화가 단순한 문서 정리가 아니라, 검색 후보의 품질과 문서 coverage를 높이는 데 직접적으로 도움을 줬다는 근거다.

해석은 다음과 같다.

- 표 row/column 추출로 표 안의 예산, 기간, 자격 정보가 검색 가능해졌다.
- `fact_candidates`가 질문에 자주 나오는 RFP 핵심 정보를 앞쪽 후보로 끌어올렸다.
- 문서명, 발주기관, 사업명 alias가 target 문서 매칭을 보강했다.
- source ref가 generation 단계에서 긴 원문을 다시 확인할 수 있는 연결고리를 만들었다.

## 7. 주요 실패 사례와 데이터 보정 방향

| 사례 | 실패 유형 | 발생 원인 | 데이터 보정 방향 |
|---|---|---|---|
| Q008 | 다문서 예산 차액 오답 | 한국수자원공사 예산이 원문에 없고 다른 금액 후보가 섞임 | 나라장터 확인값 `2,392,940,000원`을 `project_budget`으로 보강 |
| Q201/Q290 | 전체 배정액 오답 | 평가 기준 금액 `2억원`이 사업예산처럼 선택됨 | `780,230,000원`을 project budget으로 분리하고, 평가 기준 금액은 다른 역할로 보호 |
| Q249/Q369 | 예산 없음 질문에서 금액 생성 | 예산이 없는 문서에 다른 금액 후보가 들어옴 | 예산 없음 상태를 명확히 표시 |
| Q157 | 장려금/수당 질문에 사업예산 답변 | 질문 속 금액 역할과 사업예산 역할이 섞임 | 금액의 역할 분류를 강화 |
| Q100/Q453 | 복합 질문 일부만 답변 | 예산, 요약, 비교가 한 문장에 섞임 | 데이터에서는 목적/대상/운영 맥락 단서가 잘 드러나도록 문서 요약과 요구사항 chunk를 보강 방향으로 분리 |
| Q356/Q416 | 고차원 질문 대응 부족 | 기술 목적, 운영 연속성, 장애 대응 맥락이 검색 결과에서 약하게 드러남 | 기술 목적, 운영 환경, 장애 대응 관련 문맥이 검색 신호로 잡히도록 metadata와 text/table 근거 정리 |

마지막 두 유형은 데이터만으로 완전히 해결되는 문제는 아니다. 다만 데이터 측면에서는 질문의 고차원 의도를 받쳐줄 수 있는 목적, 운영 맥락, 기술 효과 관련 근거가 검색 가능한 형태로 남아 있어야 한다. 이후 generation에서는 이 근거를 intent별로 조립해야 한다.

## 8. 최종 corpus 구성과 정합성

최종 공유 corpus는 Chroma 적재와 검색 실험에 바로 사용할 수 있는 실행용 corpus 기준이다.

| 항목 | 값 |
|---|---:|
| 문서 수 | 690 |
| 기본 실행용 청크 수 | 106,777 |
| 원문 저장소 row 수 | 79,238 |
| 기본 실행용 파일 크기 | 366.61 MiB / 384.42 MB |
| 검수용 full 파일 크기 | 476.89 MiB / 500.05 MB |
| `chunk_id` 중복 | 0 |
| `source_store_id` 중복 | 0 |
| `source_ref` 누락 | 0 |
| 동일 본문 청크 중복률 | 약 4.35% |

최종 corpus는 `embed_enabled=True`인 청크만 기본 검색 대상으로 남겼다. `source_store_v2_690.jsonl`은 검색 대상이 아니라, 검색된 청크의 원문 근거를 다시 확인할 때 사용하는 보조 파일이다.

## 9. 검색 실험 요약

검색 실험은 KoE5 의미 검색, BM25, RRF, 재정렬 모델 조합을 비교했다.

| 조건 | 방식 | 해석 |
|---|---|---|
| J0 dense | KoE5 의미 검색 | 기본 기준선 |
| J2 BM25 | 키워드 검색 | 기관명, 공고명, 숫자 검색 보완 |
| J3 RRF | 의미 검색 + BM25 결합 | 속도와 품질 균형이 좋음 |
| J5 hybrid | 의미 검색 + BM25 + 재정렬 | 최종 품질이 가장 좋음 |

주요 수치는 아래와 같다.

| 기준 | 설명 | Hit@5 | MRR@5 | nDCG@5 | Doc Recall@5 | Multi-doc Recall@5 | Latency |
|---|---|---:|---:|---:|---:|---:|---:|
| v1 초기 | KoE5 dense baseline | 0.942 | 0.867 | 0.790 | 0.806 | 0.635 | avg 28.69ms |
| v2 초기 | dense + BM25 RRF + reranker | 0.980 | 0.958 | 0.886 | 0.878 | 0.750 | avg 3263.32ms |
| v2 최고 | dense + BM25 RRF + reranker | 0.990 | 0.959 | 0.9578 | 0.9825 | 0.9792 | P95 3602.69ms |

J5가 품질 기준으로 가장 좋지만, 속도까지 고려하면 J3도 현실적인 후보가 될 수 있다.

## 10. 답변 생성 설계

답변 생성 단계에서는 검색 top-k를 그대로 LLM에 넣지 않았다. 질문 유형에 따라 근거를 다시 고르고, 숫자 계산은 코드가 처리하도록 했다.

```text
검색 결과
→ chunk metadata 재결합
→ 질문 intent 분류
→ target 문서 확인
→ fact/table/text 근거 재정렬
→ 예산 차액/합계 같은 값은 코드로 계산
→ LLM은 근거와 계산 결과를 문장화
→ 후처리로 JSON, 숫자, citation, intent 누락 검사
```

예를 들어 “A 사업과 B 사업의 예산 차액은?”이라는 질문은 다음처럼 처리했다.

```text
1. 두 문서 각각의 project_budget 추출
2. candidate_only, notice_base_amount, threshold_budget 제외
3. 차액은 코드로 계산
4. LLM은 계산 결과를 자연스러운 문장으로 정리
5. citation은 LLM이 만들지 않고 코드가 근거 chunk에서 부착
```

이 구조는 LLM이 계산을 틀리거나, 후보 금액을 사업예산처럼 답하는 문제를 줄이기 위한 것이다.

Gemini는 API 모델답게 형식 안정성과 속도가 좋았다. 다만 프로젝트 보고서에서는 Gemini 자체의 우수성보다, API 모델을 품질 상한선으로 두고 HF 모델을 어디까지 안정화했는지를 보여주는 것이 더 의미 있다.

허깅페이스 모델 중에서는 Qwen2.5-3B가 가장 안정적인 기준선이었다. Qwen3-4B는 모델 자체 성능은 높을 수 있지만, 이번 RFP 한국어 RAG 작업에서는 프롬프트 유출, 과한 답변, 문맥 덤프 같은 문제가 생겨 단순 프롬프트부터 다시 잡는 실험군으로 보는 것이 안전하다.

## 12. QLoRA SFT 참고 실험

Qwen 계열 소형 모델을 프로젝트 답변 형식에 맞추기 위한 QLoRA SFT smoke 실험도 진행했다.

```text
covered_question_count = 584
train = 533
valid = 17
test = 17
hard_eval = 8
```

다만 현재 단계에서 QLoRA는 RAG context를 대체할 수 있는 수준으로 보기는 어렵다. 금액, 기간, 마감일 같은 고정값은 모델에 외우게 하기보다 retrieval context, metadata, deterministic 계산 레이어를 우선하는 것이 맞다. QLoRA는 출력 형식과 답변 스타일을 안정화하는 보조 실험으로 보는 것이 적절하다.

## 14. 결론

이번 작업은 단순 전처리가 아니라 RFP 문서를 RAG가 이해할 수 있는 형태로 바꾸는 작업이었다. v1에서 v2로 넘어가며 표 추출과 JSON key 구조화를 적용했고, 그 결과 nDCG@5, Doc Recall@5, Multi-doc Recall@5가 모두 크게 개선됐다.

특히 Multi-doc Recall@5가 0.635에서 0.9792로 오른 것은 구조화가 복합 질문 검색에 긍정적으로 작용했다는 중요한 근거다. RFP 질문은 하나의 문서만 보는 경우보다 여러 문서의 예산, 목적, 성격을 함께 비교하는 경우가 많기 때문에 이 개선은 실제 답변 생성에도 의미가 있다.

답변 생성에서는 모델만 바꾸는 것으로는 충분하지 않았다. 금액과 기간 같은 값은 코드와 metadata로 안정화하고, LLM은 근거를 바탕으로 자연스럽게 문장화하는 역할로 제한하는 방향이 더 안정적이었다.

최종적으로 API 모델인 Gemini는 품질 상한선과 속도 비교군으로, Qwen2.5-3B는 허깅페이스 모델 기반 안정판으로, Qwen3-4B는 추가 프롬프트 실험군으로 정리할 수 있다. 현재 보고서의 핵심 메시지는 “구조화된 데이터가 검색 품질을 끌어올렸고, 그 구조를 답변 생성 단계에서 활용하면서 답변 안정성을 개선했다”는 것이다.
